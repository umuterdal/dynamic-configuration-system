1. Proje Genel Bakışı

Bu proje, uygulama restart veya deployment gerektirmeden konfigürasyon değişikliklerinin runtime'da dinamik olarak yönetilmesini sağlayan production-ready bir .NET 8 sistemidir.

Temel Özellikler

  • Dinamik Konfigürasyon: Deployment veya restart olmadan runtime güncellemeler
  • Service Isolation: Her servis sadece kendi konfigürasyon kayıtlarını görür
  • Tip Güvenliği: GetValue<T>() ile generic, strongly-typed erişim
  • Thread Güvenliği: ConcurrentDictionary ile eşzamanlı erişim
  • Otomatik Yenileme: BackgroundService + PeriodicTimer (30sn varsayılan)
  • Graceful Degradation: Storage erişilemezse son başarılı cache ile çalışmaya devam
  • Message Broker: RabbitMQ ile anlık cache yenileme (polling'i tamamlar)
  • Otomatik Seed Data: İlk çalıştırmada demo verileri otomatik eklenir

5. ConfigurationReader

Kütüphanenin dışarıya açık API'si. 3 parametre ile initialize edilir:

var reader = new ConfigurationReader(
    applicationName: "SERVICE-A",
    connectionString: "mongodb://localhost:27017",
    refreshTimerIntervalInMs: 30000);

Parametreler

ParametreAçıklama
applicationNameUygulama adı. Servis izolasyonu için kullanılır.
connectionStringMongoDB bağlantı adresi.
refreshTimerIntervalInMsPeriyodik yenileme aralığı (milisaniye).

Public Methodlar

T GetValue<T>(string key)

Belirtilen key için değeri tip güvenli olarak döner. Supports: string, int, double, bool.

string siteName = reader.GetValue<string>("SiteName");      // "soty.io"
int maxItems = reader.GetValue<int>("MaxItemCount");        // 500
double taxRate = reader.GetValue<double>("TaxRate");        // 0.18
bool isEnabled = reader.GetValue<bool>("IsBasketEnabled"); // true

bool TryGetValue<T>(string key, out T? value)

Key mevcutsa değeri döner, yoksa false. Exception fırlatmaz.

if (reader.TryGetValue<string>("SiteName", out var name))
    Console.WriteLine(name);

IReadOnlyDictionary<string, string> GetAllValues()

Tüm aktif konfigürasyonları dictionary olarak döner.

Task RefreshAsync()

Cache'i manuel olarak yeniler. Eşzamanlı çağrılarda coalesce edilir (interlocked guard).

object GetHealthInfo()

Cache durumu hakkında bilgi döner: son yenileme zamanı, cache yaşı, polling aralığı, cache'lenen key'ler.

Önemli ConfigurationReader constructor'da MongoDB'ye bağlanıp ilk yükleme yapar. Bu nedenle constructor synchronous olarak LoadConfigurationsAsync().GetAwaiter().GetResult() çağırır.

6. Cache Mekanizması

Neden ConcurrentDictionary?

  • O(1) ortalama okuma süresi
  • Thread-safe — built-in senkronizasyon
  • Lock-free reads — çoğu operasyonda kilit yok
  • Sıfır dış bağımlılık — MemoryCache'e gerek yok

Cache Güncelleme Stratejisi

Yenileme sırasında atomic swap kullanılır:

// 1. Yeni cache oluştur
var newCache = new ConcurrentDictionary<string, ConfigurationRecord>();

// 2. Veritabanından taze verileri yükle
foreach (var record in records)
    newCache[record.Name] = record;

// 3. Atomic swap — referansı değiştir, kilit yok
Interlocked.Exchange(ref _cache, newCache);

Eşzamanlılık Güvenliği

  • Single writer: Yenileme işlemini sadece bir BackgroundService yapar
  • Multiple readers: Eşzamanlı okuma işlemleri ConcurrentDictionary tarafından yönetilir
  • Overlapping refresh prevention: Interlocked.CompareExchange ile eşzamanlı yenilemeler engellenir

Case-Insensitive Key Erişimi

Cache, StringComparer.OrdinalIgnoreCase ile oluşturulur. Bu sayede "sitename" ve "SiteName" aynı key olarak tanınır.

7. Periyodik Yenileme (Polling)

ConfigurationRefreshService, BackgroundService kullanarak periyodik olarak cache'i yeniler.

Neden BackgroundService + PeriodicTimer?

YaklaşımArtılarıEksileri
BackgroundService + PeriodicTimer SeçilenBuilt-in yaşam döngüsü, graceful shutdown, DI entegrasyonu—
System.Threading.TimerDaha fazla kontrolOrphaned callback riski, manuel yaşam döngüsü
HangfireDashboard, persistenceOver-engineered, ek bağımlılık

Implementasyon

protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    using var timer = new PeriodicTimer(_refreshInterval);

    while (await timer.WaitForNextTickAsync(stoppingToken))
    {
        try
        {
            await _configurationReader.RefreshAsync();
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Refresh failed");
        }
    }
}

Polling Nasıl Test Edilir?

  1. Cache durumunu kontrol et: curl http://localhost:5001/api/demo/health
  2. Admin Panel'den değeri değiştir (ör: SiteName: "soty.io" → "yeni-site.com")
  3. Cache hemen değişmez (30sn polling süresi var)
  4. Manuel refresh ile anında test et: curl -X POST http://localhost:5001/api/demo/refresh
  5. Veya 30sn bekle → otomatik yenilenir

8. Message Broker (RabbitMQ)

RabbitMQ, polling'i tamamlar, yerine geçmez. polling ana mekanizmadır, broker sadece gecikmeyi azaltır.

Neden Her İki Mekanizma?

MekanizmaRolüGecikme
PollingAna mekanizma — her 30sn'de bir kontrol0-30sn
RabbitMQİyileştirme — değişiklik anında bildirim~100ms

Neden İkisi Birden?

  • Polling tek başına yeterli — broker çökse bile sistem çalışır
  • Broker tek başına yetersiz — bağlantı koparsa polling kesintisiz devam eder
  • Production gerçeği: Hiçbir message broker %100 güvenilir değildir. Polling fallback olarak her zaman çalışır.

Mimari

  • Publisher: RabbitMqConfigurationBrokerPublisher — Admin CRUD sonrası change event'i publish eder
  • Consumer: ConfigurationBrokerConsumer — Event'i alır, anında cache refresh yapar
  • Exchange: Fanout exchange — tüm consumer'lara bildirim gider
  • Ack: Manuel acknowledgment — message işlendikten sonra onaylanır

Graceful Degradation

RabbitMQ çökse bile:

  • Polling devam eder
  • Publisher hatası uygulamayı bozmaz (try-catch ile sarılmış)
  • Consumer bağlantı hatası loglanır, polling fallback olarak çalışır

// JavaScript — keyup event ile Name sütununda filtreleme

    var filter = this.value.toLowerCase();
    var rows = document.querySelectorAll('#configurationTable tr');
    rows.forEach(function (row) {
        if (row.cells.length < 6) return;
        var name = row.cells[1]?.textContent.toLowerCase() || '';
        row.style.display = name.includes(filter) ? '' : 'none';
    });
});

14. Hata Yönetimi

Graceful Degradation

Storage erişilemezse sistem son başarılı cache ile çalışmaya devam eder:

  1. İlk yükleme başarısız: Boş cache — uygulama başlar ama veri yok
  2. Yenileme başarısız: Önceki cache korunur
  3. Uygulama çalışmaya devam eder — eski verilerle
  4. Warning logları — monitoring için

Implementasyon

private async Task RefreshConfigurationsAsync()
{
    try
    {
        var records = await _repository.GetByApplicationNameAsync(_applicationName);
        var newCache = new ConcurrentDictionary<string, ConfigurationRecord>();
        foreach (var record in records)
            newCache[record.Name] = record;
        Interlocked.Exchange(ref _cache, newCache);
    }
    catch (Exception ex)
    {
        // HATA: Cache'deki önceki değerleri KORU
        _logger.LogWarning(ex,
            "Failed to refresh. Cache will retain previous values.");
    }
}

RabbitMQ Hata Yönetimi

  • Publisher: Hata durumunda exception fırlatmaz, log yazar
  • Consumer: BasicNack ile mesajı tekrar kuyruğa atar
  • Bağlantı kopması: AutomaticRecoveryEnabled = true ile otomatik yeniden bağlanma

Exponential Backoff

Yenileme denemelerinde üstel gecikme stratejisi kullanılır:

  • İlk deneme: anında
  • 2. deneme: 500ms bekleme
  • 3. deneme: 1000ms bekleme
  • Maksimum 3 deneme — başarısız olursa cache korunur
